---
title: "Local Deployment"
description: "Set up a local Feast deployment for development using file-based offline store and SQLite online store. No cloud credentials required."
---

The local deployment mode is the fastest way to get Feast running on your machine. It uses local Parquet files as the offline store and SQLite as the online store — no cloud accounts or external services needed.

## Quickstart

<Steps>
  <Step title="Install Feast">
    Install the Feast SDK with pip:

    ```bash
    pip install feast
    ```
  </Step>

  <Step title="Initialize a feature repository">
    The `feast init` command scaffolds a new feature repository with sample data, feature definitions, and a local `feature_store.yaml`:

    ```bash
    feast init my_feature_repo
    cd my_feature_repo/feature_repo
    ```

    The generated directory looks like:

    ```
    my_feature_repo/
    └── feature_repo/
        ├── data/
        │   └── driver_stats.parquet
        ├── example_repo.py
        └── feature_store.yaml
    ```
  </Step>

  <Step title="Review the local feature_store.yaml">
    The default `feature_store.yaml` for local development uses the `local` provider, a local registry file, and SQLite as the online store:

    ```yaml
    project: my_feature_repo
    registry: data/registry.db
    provider: local
    online_store:
      type: sqlite
      path: data/online_store.db
    entity_key_serialization_version: 2
    ```

    | Field | Description |
    |---|---|
    | `project` | Namespace that isolates this feature store from others |
    | `registry` | Path to the local SQLite registry file |
    | `provider` | `local` — uses pandas for point-in-time joins and SQLite for online serving |
    | `online_store.type` | `sqlite` — lightweight file-based online store |
    | `online_store.path` | Path to the SQLite database file |
  </Step>

  <Step title="Apply feature definitions">
    Register all feature views, entities, and feature services defined in your Python files with the feature store:

    ```bash
    feast apply
    ```

    Expected output:

    ```
    Applying changes for project my_feature_repo

    Created entity driver
    Created feature view driver_hourly_stats
    Created feature view driver_hourly_stats_fresh
    Created feature service driver_activity_v1

    Created sqlite table my_feature_repo_driver_hourly_stats
    Created sqlite table my_feature_repo_driver_hourly_stats_fresh
    ```

    <Warning>
      `feast apply` registers feature definitions and creates infrastructure (e.g. SQLite tables), but does **not** load any data into the online store. Run `feast materialize` to populate the online store.
    </Warning>
  </Step>

  <Step title="Materialize features into the online store">
    Load historical feature data from the offline store (local Parquet files) into the online store (SQLite):

    ```bash
    # Materialize a specific time range
    feast materialize 2021-04-07T00:00:00 2021-04-08T00:00:00

    # Or materialize only new data since the last run (recommended for recurring jobs)
    feast materialize-incremental $(date -u +"%Y-%m-%dT%H:%M:%S")
    ```

    You should see output like:

    ```
    Materializing 2 feature views to 2021-09-09 17:00:00 into the sqlite online store.

    driver_hourly_stats from 2021-09-09 16:51:08 to 2021-09-09 17:00:00:
    100%|████████████████████████| 5/5 [00:00<00:00, 295.24it/s]
    ```

    <Info>
      `materialize-incremental` tracks the last materialized timestamp in the registry, so subsequent runs only process new data. Use `materialize` when you need precise control over the time window.
    </Info>
  </Step>

  <Step title="Start the feature server">
    Launch the HTTP feature server to serve features over a REST API:

    ```bash
    feast serve
    ```

    The server starts on port `6566` by default:

    ```
    INFO: Uvicorn running on http://127.0.0.1:6566 (Press CTRL+C to quit)
    ```

    Test it with a quick curl request:

    ```bash
    curl -X POST "http://localhost:6566/get-online-features" \
      -H "Content-Type: application/json" \
      -d '{
        "features": [
          "driver_hourly_stats:conv_rate",
          "driver_hourly_stats:acc_rate"
        ],
        "entities": {
          "driver_id": [1001, 1002]
        }
      }'
    ```
  </Step>
</Steps>

## Complete local feature_store.yaml reference

Below is a fully annotated `feature_store.yaml` for a local development setup:

```yaml
# feature_store.yaml (local development)
project: my_project                   # Logical namespace for the feature store
registry: data/registry.db            # Local SQLite registry — stores feature metadata
provider: local                       # Use pandas for offline queries, SQLite for online

online_store:
  type: sqlite
  path: data/online_store.db          # SQLite database for low-latency online retrieval

offline_store:
  type: file                          # Read from local Parquet files (default for local provider)

# Optional: feature server configuration
feature_server:
  type: local

entity_key_serialization_version: 2
```

## CLI command reference

| Command | Description |
|---|---|
| `feast init [name]` | Scaffold a new feature repository |
| `feast apply` | Register feature definitions and create infrastructure |
| `feast materialize <start> <end>` | Load features for a specific time range into the online store |
| `feast materialize-incremental <end>` | Load only new features since the last run |
| `feast serve` | Start the HTTP feature server on port 6566 |
| `feast serve --port 8080` | Start on a custom port |
| `feast ui` | Open the Feast web UI |
| `feast teardown` | Remove all Feast-managed infrastructure |

<Warning>
  `feast teardown` is irreversible. It removes all tables and infrastructure managed by Feast. Proceed with caution.
</Warning>

## Retrieving features in Python

You can also retrieve features directly via the Python SDK without starting the HTTP server:

```python
from feast import FeatureStore

store = FeatureStore(repo_path=".")

feature_vector = store.get_online_features(
    features=[
        "driver_hourly_stats:conv_rate",
        "driver_hourly_stats:acc_rate",
        "driver_hourly_stats:avg_daily_trips",
    ],
    entity_rows=[
        {"driver_id": 1001},
        {"driver_id": 1002},
    ],
).to_dict()

print(feature_vector)
```
