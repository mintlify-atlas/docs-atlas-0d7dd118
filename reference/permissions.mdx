---
title: "Permissions & RBAC"
description: "Reference for Feast's role-based access control (RBAC) system: the Permission class, AuthzedAction, policy types, and authentication configuration."
---

Feast includes a permissions framework that lets you control which users or roles can perform which operations on which feature store objects. Permissions are defined as code (like any other Feast object), registered with `store.apply()`, and enforced by the Feast feature server.

```python
from feast.permissions.permission import Permission
from feast.permissions.action import AuthzedAction
from feast.permissions.policy import RoleBasedPolicy
```

---

## AuthzedAction

`AuthzedAction` is an enum that identifies the type of operation being secured. Permissions reference one or more actions to specify what they protect.

```python
from feast.permissions.action import AuthzedAction, ALL_ACTIONS, READ, WRITE, CRUD
```

| Action | Value | Description |
|---|---|---|
| `AuthzedAction.CREATE` | `"create"` | Create a new object instance |
| `AuthzedAction.DESCRIBE` | `"describe"` | Read the object's definition/state |
| `AuthzedAction.UPDATE` | `"update"` | Modify an existing object |
| `AuthzedAction.DELETE` | `"delete"` | Delete an object |
| `AuthzedAction.READ_ONLINE` | `"read_online"` | Read from the online store |
| `AuthzedAction.READ_OFFLINE` | `"read_offline"` | Read from the offline store |
| `AuthzedAction.WRITE_ONLINE` | `"write_online"` | Write to the online store |
| `AuthzedAction.WRITE_OFFLINE` | `"write_offline"` | Write to the offline store |

**Convenience aliases:**

```python
ALL_ACTIONS  # All 8 actions above
READ         # [READ_OFFLINE, READ_ONLINE]
WRITE        # [WRITE_OFFLINE, WRITE_ONLINE]
CRUD         # [CREATE, DESCRIBE, UPDATE, DELETE]
```

## Policy Types

A `Policy` determines how to validate a user against the permission. Feast includes four built-in policy implementations.

<Accordion title="RoleBasedPolicy">

```python
RoleBasedPolicy(roles: list[str])
```

Grants access when the authenticated user has **at least one** of the specified roles. Roles are extracted from the user's JWT token (OIDC) or Kubernetes service account.

<ParamField path="roles" type="list[str]" required>
  One or more role names. The user must hold at least one to satisfy the policy.
</ParamField>

```python
from feast.permissions.policy import RoleBasedPolicy

# User must have either "admin" or "ml_engineer"
policy = RoleBasedPolicy(roles=["admin", "ml_engineer"])
```

</Accordion>

<Accordion title="GroupBasedPolicy">

```python
GroupBasedPolicy(groups: list[str])
```

Grants access when the authenticated user belongs to **at least one** of the specified groups.

<ParamField path="groups" type="list[str]" required>
  One or more group names.
</ParamField>

```python
from feast.permissions.policy import GroupBasedPolicy

policy = GroupBasedPolicy(groups=["data-science-team", "platform-team"])
```

</Accordion>

<Accordion title="NamespaceBasedPolicy">

```python
NamespaceBasedPolicy(namespaces: list[str])
```

Grants access when the authenticated user is associated with **at least one** of the specified Kubernetes namespaces.

<ParamField path="namespaces" type="list[str]" required>
  One or more Kubernetes namespace names.
</ParamField>

```python
from feast.permissions.policy import NamespaceBasedPolicy

policy = NamespaceBasedPolicy(namespaces=["production", "staging"])
```

</Accordion>

<Accordion title="CombinedGroupNamespacePolicy">

```python
CombinedGroupNamespacePolicy(groups: list[str], namespaces: list[str])
```

Grants access when the user is in **at least one** of the specified groups **or** at least one of the specified namespaces (logical OR).

<ParamField path="groups" type="list[str]" required>
  Permitted group names.
</ParamField>

<ParamField path="namespaces" type="list[str]" required>
  Permitted Kubernetes namespace names.
</ParamField>

```python
from feast.permissions.policy import CombinedGroupNamespacePolicy

policy = CombinedGroupNamespacePolicy(
    groups=["admin-group"],
    namespaces=["production"],
)
```

</Accordion>

<Accordion title="AllowAll (no restriction)">

The `AllowAll` singleton is the default policy when no `policy` argument is provided to `Permission`. It grants access to all users unconditionally.

```python
from feast.permissions.policy import AllowAll

# These two are equivalent:
Permission(name="open", actions=AuthzedAction.DESCRIBE, policy=AllowAll)
Permission(name="open", actions=AuthzedAction.DESCRIBE)
```

</Accordion>

## Permission Class

<Accordion title="Permission.__init__">

```python
Permission(
    name: str,
    types: Optional[Union[list[FeastObject], FeastObject]] = [],
    name_patterns: Optional[Union[str, list[str]]] = [],
    actions: Union[list[AuthzedAction], AuthzedAction] = ALL_ACTIONS,
    policy: Policy = AllowAll,
    tags: Optional[dict[str, str]] = None,
    required_tags: Optional[dict[str, str]] = None,
)
```

Defines a permission rule. When a request arrives at the feature server, Feast evaluates all registered permissions whose `types`, `name_patterns`, `required_tags`, and `actions` match the request, then validates the caller against each matching permission's `policy`.

<ParamField path="name" type="str" required>
  A human-readable name for this permission. Names can be duplicated (used for logging/troubleshooting only; the object is identified by identity, not name).
</ParamField>

<ParamField path="types" type="Optional[Union[list[FeastObject], FeastObject]]">
  The Feast resource types this permission applies to. Matching is inclusive of all subclasses. Defaults to `ALL_RESOURCE_TYPES` when empty or `None`.

  <Expandable title="Manageable resource types">
    | Type class | Description |
    |---|---|
    | `feast.project.Project` | Project definitions |
    | `feast.feature_view.FeatureView` | Standard batch feature views |
    | `feast.on_demand_feature_view.OnDemandFeatureView` | On-demand feature views |
    | `feast.batch_feature_view.BatchFeatureView` | Batch feature views |
    | `feast.stream_feature_view.StreamFeatureView` | Stream feature views |
    | `feast.entity.Entity` | Entity definitions |
    | `feast.feature_service.FeatureService` | Feature services |
    | `feast.data_source.DataSource` | Data source definitions |
    | `feast.saved_dataset.ValidationReference` | Validation references |
    | `feast.saved_dataset.SavedDataset` | Saved datasets |
    | `feast.permissions.permission.Permission` | Permission objects themselves |
  </Expandable>
</ParamField>

<ParamField path="name_patterns" type="Optional[Union[str, list[str]]]">
  One or more regex patterns to filter resources by name. Only resources whose names match **at least one** pattern are subject to this permission. Defaults to empty list (match all names).
</ParamField>

<ParamField path="actions" type="Union[list[AuthzedAction], AuthzedAction]">
  The actions this permission secures. Defaults to `ALL_ACTIONS`.
</ParamField>

<ParamField path="policy" type="Policy">
  The policy to evaluate against the requesting user. Defaults to `AllowAll`.
</ParamField>

<ParamField path="tags" type="Optional[dict[str, str]]">
  Metadata tags stored on the permission object itself.
</ParamField>

<ParamField path="required_tags" type="Optional[dict[str, str]]">
  A dictionary of key-value pairs that a resource **must** have in its own `tags` for this permission to apply. Permissions with unmatched `required_tags` are skipped for that resource.
</ParamField>

</Accordion>

## Registering Permissions

Permissions are Feast objects. Register them with `store.apply()` alongside your entities and feature views:

```python
from feast import FeatureStore, FeatureView
from feast.permissions.permission import Permission
from feast.permissions.action import AuthzedAction, READ, WRITE
from feast.permissions.policy import RoleBasedPolicy

store = FeatureStore(repo_path="feature_repo/")

# Allow the "analyst" role to read all feature views
analyst_read = Permission(
    name="analyst_read_fvs",
    types=[FeatureView],
    actions=READ,
    policy=RoleBasedPolicy(roles=["analyst"]),
)

# Allow the "mlops" role full access to all objects
mlops_admin = Permission(
    name="mlops_full_access",
    actions=list(AuthzedAction),
    policy=RoleBasedPolicy(roles=["mlops"]),
)

store.apply([analyst_read, mlops_admin])
```

## Examples

<Accordion title="Restrict a specific feature view to a role">

Use `name_patterns` to match a specific feature view name and restrict write access.

```python
from feast.permissions.permission import Permission
from feast.permissions.action import AuthzedAction
from feast.permissions.policy import RoleBasedPolicy
from feast.feature_view import FeatureView

# Only "data_engineer" can write to any feature view whose name starts with "driver_"
permission = Permission(
    name="driver_fv_write_restriction",
    types=[FeatureView],
    name_patterns=["driver_.*"],
    actions=[
        AuthzedAction.WRITE_ONLINE,
        AuthzedAction.WRITE_OFFLINE,
    ],
    policy=RoleBasedPolicy(roles=["data_engineer"]),
)

store.apply([permission])
```

</Accordion>

<Accordion title="Protect only tagged resources">

Use `required_tags` to apply a permission only to resources that carry specific metadata.

```python
from feast.permissions.permission import Permission
from feast.permissions.action import AuthzedAction
from feast.permissions.policy import RoleBasedPolicy
from feast.feature_view import FeatureView

# Only "pii_steward" can read feature views tagged as "pii: true"
pii_read = Permission(
    name="pii_feature_view_read",
    types=[FeatureView],
    actions=[AuthzedAction.READ_ONLINE, AuthzedAction.READ_OFFLINE],
    policy=RoleBasedPolicy(roles=["pii_steward"]),
    required_tags={"pii": "true"},
)

store.apply([pii_read])
```

Then tag the protected feature views:

```python
sensitive_fv = FeatureView(
    name="user_sensitive_data",
    ...
    tags={"pii": "true"},
)
```

</Accordion>

<Accordion title="Online store access by group">

```python
from feast.permissions.policy import GroupBasedPolicy
from feast.permissions.action import AuthzedAction
from feast.permissions.permission import Permission
from feast.feature_service import FeatureService

# Allow the "inference-service" group to call get_online_features
inference_permission = Permission(
    name="inference_service_online_read",
    types=[FeatureService],
    actions=[AuthzedAction.READ_ONLINE],
    policy=GroupBasedPolicy(groups=["inference-service"]),
)

store.apply([inference_permission])
```

</Accordion>

## Authentication Configuration

Feast supports three auth backends configured in `feature_store.yaml`. Permissions are only enforced when a non-`no_auth` backend is active.

<Accordion title="No Auth (default)">

```yaml
# feature_store.yaml
auth:
  type: no_auth
```

No authentication. All requests are allowed. Suitable for local development.

</Accordion>

<Accordion title="OIDC Authentication">

Feast validates a Bearer token from the `Authorization` HTTP header against an OIDC discovery endpoint. Roles are extracted from the token claims.

```yaml
# feature_store.yaml
auth:
  type: oidc
  auth_discovery_url: https://accounts.google.com/.well-known/openid-configuration
  client_id: my-feast-client
```

**Client configuration** (`OidcClientAuthConfig`) for SDK clients connecting to a remote feature server:

```python
from feast.permissions.auth_model import OidcClientAuthConfig

# Pre-issued token
auth_config = OidcClientAuthConfig(
    type="oidc",
    auth_discovery_url="https://idp.example.com/.well-known/openid-configuration",
    client_id="feast-client",
    token="eyJhbGci...",  # pre-issued token
)

# Client credentials flow
auth_config = OidcClientAuthConfig(
    type="oidc",
    auth_discovery_url="https://idp.example.com/.well-known/openid-configuration",
    client_id="feast-client",
    client_secret="s3cr3t",
)

# Resource Owner Password Grant (ROPG)
auth_config = OidcClientAuthConfig(
    type="oidc",
    auth_discovery_url="https://idp.example.com/.well-known/openid-configuration",
    client_id="feast-client",
    username="alice",
    password="hunter2",
    client_secret="s3cr3t",
)
```

</Accordion>

<Accordion title="Kubernetes Authentication">

Feast uses Kubernetes service account tokens for server-side authentication, and optionally accepts a user token for non-service-account callers. Roles are derived from Kubernetes RBAC.

```yaml
# feature_store.yaml
auth:
  type: kubernetes
```

**Client configuration** for SDK clients:

```python
from feast.permissions.auth_model import KubernetesAuthConfig

# Use the pod's mounted service account token (default)
auth_config = KubernetesAuthConfig(type="kubernetes")

# Supply an explicit user token
auth_config = KubernetesAuthConfig(
    type="kubernetes",
    user_token="eyJhbGci...",
)
```

Use `NamespaceBasedPolicy` or `CombinedGroupNamespacePolicy` to control access by Kubernetes namespace:

```python
from feast.permissions.policy import NamespaceBasedPolicy
from feast.permissions.permission import Permission
from feast.permissions.action import AuthzedAction

# Restrict materialization to callers in the "mlops" namespace
materialize_perm = Permission(
    name="production_materialize",
    actions=[AuthzedAction.WRITE_ONLINE],
    policy=NamespaceBasedPolicy(namespaces=["mlops"]),
)

store.apply([materialize_perm])
```

</Accordion>

## Full Example: RBAC Feature Repository

```python
# feature_repo/permissions.py

from feast.feature_view import FeatureView
from feast.feature_service import FeatureService
from feast.entity import Entity
from feast.permissions.permission import Permission
from feast.permissions.action import (
    AuthzedAction, ALL_ACTIONS, READ, WRITE, CRUD,
)
from feast.permissions.policy import RoleBasedPolicy

# Analysts can only read feature definitions and retrieve from online store
analyst_permission = Permission(
    name="analyst",
    types=[FeatureView, FeatureService],
    actions=READ + [AuthzedAction.DESCRIBE],
    policy=RoleBasedPolicy(roles=["analyst"]),
)

# Data engineers can manage feature definitions but not serve online
data_engineer_permission = Permission(
    name="data_engineer",
    types=[FeatureView, Entity, FeatureService],
    actions=CRUD + [AuthzedAction.WRITE_OFFLINE],
    policy=RoleBasedPolicy(roles=["data_engineer"]),
)

# Admins have full access
admin_permission = Permission(
    name="admin",
    actions=ALL_ACTIONS,
    policy=RoleBasedPolicy(roles=["admin"]),
)
```

```python
# feature_repo/__init__.py or apply script

from feast import FeatureStore
from feature_repo.permissions import (
    analyst_permission,
    data_engineer_permission,
    admin_permission,
)

store = FeatureStore(repo_path="feature_repo/")
store.apply([analyst_permission, data_engineer_permission, admin_permission])
```

<Note>
Permissions are evaluated server-side. They have no effect when the `FeatureStore` is used locally without a feature server (e.g., in notebooks or unit tests).
</Note>
